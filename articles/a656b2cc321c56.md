## git worktreeによる並列開発時の悩み
claude codeに実装してもらうようになり、git worktreeを作って、複数のタスクを同時にこなしている方は多いのではないでしょうか？
Android開発において、git worktreeを使うと次のような悩みが発生してしまいます。
「worktreeを作ったばかりのときにビルドをすると遅い」
ビルドキャッシュはworktreeごとに生成されるため、worktreeを作成した直後はビルドキャッシュが存在せず、ビルド時間が長くなってしまいます。
これではせっかく並列開発できても、イテレーションを高速に回せません。

## 結論: dockerでリモートビルドキャッシュサーバーを立てることでビルド時間が改善できた！
先に結論をお話しますが、dockerでリモートビルドキャッシュサーバーを立てることで、worktreeでのビルド時間を1/7に改善することができました。
before: 約13分
after: 約1分
以降では、具体的な手順と実験結果について説明します。

## How to

### 1. gradle/build-cache-nodeのインスタンスを立てる
[gradle/build-cache-node](https://hub.docker.com/r/gradle/build-cache-node/)は、gradle公式が提供しているリモートビルドキャッシュサーバーのdockerイメージです。
このインスタンスを立てることで、簡単にリモートビルドキャッシュサーバーを手元に構築できます。

```bash
docker run -d \
  --name gradle-cache \
  -p 5071:5071 \
  -v $HOME/gradle-build-cache:/data \
  gradle/build-cache-node
```

コンテナを起動したらログを確認し、認証情報を取得します。

```bash
$ docker logs gradle-cache | grep -o "username and password: [^ ]* [^ ]*" | tail -1
username and password: xxxx yyyyyyyyyyyyyyyyyyyy
```

http://localhost:5071 にアクセスし、上記で取得した認証情報でログインすることで、ビルドキャッシュサーバーの管理画面にアクセスできます。

デフォルトの状態では、認証情報を使わないと、キャッシュアクセスができません。
本記事ではローカル開発環境での利用を想定するので、認証なしでもキャッシュアクセス可能にします。

```bash
cd $HOME/gradle-build-cache
mkdir conf
cat > conf/config.yaml << 'YAML'
version: 5
cache:
  accessControl:
    anonymousLevel: readwrite
YAML
```

### 2. settings.gradle.ktsにリモートビルドキャッシュの設定を追加する
つづいて、プロジェクトのsettings.gradle.ktsにリモートビルドキャッシュの設定を追加します。

```kotlin
buildCache {
    local {
        isEnabled = true
    }
    remote<HttpBuildCache> {
        url = uri("http://localhost:5071/cache/")
        isAllowInsecureProtocol = true // httpを使う場合に必要
        isPush = true // ローカルキャッシュをリモートに保存する
    }
}
```

### 3. worktreeを作成して並列ビルドする
最後にworktreeを複数作成し、ビルドやテストを並列で実行します。

```bash
git worktree add ../feature/awesome-feature-1 feature/awesome-feature-1
cd ../feature/awesome-feature
./gradlew assembleDebug
```

```bash
git worktree add ../feature/awesome-feature-2 feature/awesome-feature-2
cd ../feature/awesome-feature-2
./gradlew assembleDebug
```

リモートビルドキャッシュが利用できる状態であれば、worktreeごとにビルドキャッシュが生成されていなくても、リモートビルドキャッシュからキャッシュを取得できるため、ビルド時間が大幅に短縮されます。

次に、実際にどれくらいビルド時間が改善されたかを実験で確認します。

## 実験: gradle-profiler でビルド時間を比較する
実験には [gradle-profiler](https://github.com/gradle/gradle-profiler) というGradle用のベンチマークツールを使用します。

シナリオは以下のとおりです

```conf
# ベースライン: キャッシュなし
no_cache {
    title = "キャッシュなし"
    tasks = ["assembleDebug"]
    cleanup-tasks = ["clean"]
    gradle-args = ["--no-build-cache"]
    warm-ups = 2
}

# ローカルキャッシュのみ
local_cache_only {
    title = "ローカルキャッシュのみ"
    tasks = ["assembleDebug"]
    cleanup-tasks = ["clean"]
    gradle-args = ["--build-cache"]
    warm-ups = 2
}

# リモートキャッシュあり
# ローカルキャッシュも有効
remote_cache {
    title = "リモートキャッシュ"
    tasks = ["assembleDebug"]
    cleanup-tasks = ["clean"]
    gradle-args = ["--build-cache"]
    warm-ups = 2
}

# ユニットテスト
unit_test_no_cache {
    title = "ユニットテスト (キャッシュなし)"
    tasks = ["testDebugUnitTest"]
    cleanup-tasks = ["clean"]
    gradle-args = ["--no-build-cache"]
    warm-ups = 1
}

unit_test_with_cache {
    title = "ユニットテスト (キャッシュあり)"
    tasks = ["testDebugUnitTest"]
    cleanup-tasks = ["clean"]
    gradle-args = ["--build-cache"]
    warm-ups = 1
}
```

例えばベースラインとなるキャッシュなしのビルドは次のように実行します

```bash
gradle-profiler --benchmark \
  --project-dir . \
  --scenario-file benchmark.scenarios \
  --output-dir profile-out-nocache \
  no_cache
```

実験結果は以下のとおりです。

### 単体ビルド: assembleDebug

| 構成 | ビルド時間 | 改善率 |
|------|-----------|--------|
| キャッシュなし | 154.6秒 | - |
| ローカルキャッシュのみ | 16.9秒 | 9.1倍 |
| リモートキャッシュ | 16.1秒 | 9.6倍 |

### 単体ビルド: testDebugUnitTest

| 構成 | ビルド時間 | 改善率 |
|------|-----------|--------|
| キャッシュなし | 433.6秒 (7.2分) | - |
| キャッシュあり | 19.2秒 | 22.6倍 |

### 並列ビルド: 4 worktree 同時実行

**assembleDebug**
| 構成 | 実時間 | 各ビルド時間 | 改善率 |
|------|--------|-------------|--------|
| ローカルキャッシュのみ | 786秒 (13分) | 各13分 | - |
| リモートキャッシュあり | 109秒 (1分49秒) | 52秒〜1分47秒 | **7.2倍** |

**testDebugUnitTest**
| 構成 | 実時間 | 各ビルド時間 | 改善率 |
|------|--------|-------------|--------|
| ローカルキャッシュのみ | 2644秒 (44分) | 各44分 | - |
| リモートキャッシュあり | 1149秒 (19分) | 10分〜19分 | **2.3倍** |

## まとめ
git worktreeを使った並列開発時におけるビルド時間の悩みを解決するために、dockerでリモートビルドキャッシュサーバーを立てる方法を紹介しました。
実験の結果、worktreeでのビルド時間を1/7に改善することができました。
リモートビルドキャッシュサーバーの構築は非常に簡単なので、ぜひ試してみてください。
キャッシュの威力を実感できたので、今後はCI環境のビルド時間短縮にも応用していきたいと考えています。
